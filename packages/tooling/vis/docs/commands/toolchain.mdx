# vis toolchain

Inspects the repository's tool pins (`engines.node`, `.nvmrc`, `.node-version`, `packageManager`, `.prototools`, `.mise.toml`, `.tool-versions`, `volta` field) and delegates to whichever version manager the developer already has installed.

Unlike [vite+](https://viteplus.dev/guide/env), which ships its own managed Node.js runtime under `~/.vite-plus`, vis does not try to be a version manager. It keeps the binary small and fits into whatever ecosystem (proto, mise, fnm, volta, asdf, nvm) you already use.

## Usage

```bash
vis toolchain <subcommand> [options]
```

## Subcommands

| Subcommand                        | Purpose                                                                                          |
| --------------------------------- | ------------------------------------------------------------------------------------------------ |
| `vis toolchain status`            | Show the detected manager plus expected-vs-actual versions for every tool pin found.             |
| `vis toolchain detect`            | Print the detected manager's name (useful for shell scripts).                                    |
| `vis toolchain install`           | Run `<manager> install` so every tool pin becomes active in this shell.                          |
| `vis toolchain use <tool>@<ver>`  | Pin a tool version through the detected manager. Writes the manager's native config file.       |
| `vis toolchain which <tool>`      | Resolve the binary path the manager would launch. Falls back to `PATH`.                          |

## Examples

```bash
# Read every pin and report drift
vis toolchain status

# Ask the detected manager to install everything the workspace pins
vis toolchain install

# Pin node 22.13.0 via whichever manager is active (writes .prototools,
# .tool-versions, .mise.toml, or package.json depending on the manager)
vis toolchain use node@22.13.0

# Where would `node` actually run from?
vis toolchain which node

# Name of the detected manager — pipe-friendly
vis toolchain detect       # prints "proto" / "mise" / "fnm" / ... / "none"
```

## Options

| Option        | Subcommand(s) | Description                                                                        |
| ------------- | ------------- | ---------------------------------------------------------------------------------- |
| `--json`      | `status`      | Emit a machine-readable JSON payload instead of the formatted status report.       |
| `--exit-code` | `status`      | Exit with code 1 when any tool mismatches. Useful in CI preflight jobs.            |
| `--dry-run`   | `install`, `use` | Print the command that would run but don't execute it.                          |

## How detection works

1. Walk `PATH` for `proto`, `mise`, `fnm`, `volta`, `asdf`. `nvm` is a shell function, so vis treats `$NVM_DIR` as the install marker.
2. Check for workspace-local config files: `.prototools`, `.mise.toml`, `.tool-versions`, `.nvmrc` / `.node-version`, or a `volta` field in `package.json`.
3. Pick the first manager that is **both installed and referenced by a config file**.
4. If nothing is referenced, pick the first installed manager.
5. If nothing is installed, pick the first manager with a matching config (so vis can still point at a sensible install).
6. Fall back to `{ name: "none" }`.

An explicit `toolchain.preferredManager` in `vis.config.ts` always wins.

## Tool pin priority

When the same tool is pinned in several places, later sources override earlier ones:

1. `engines.node` in `package.json`
2. `packageManager` in `package.json`
3. `volta.<tool>` in `package.json`
4. `.nvmrc` / `.node-version`
5. `.tool-versions`
6. `.mise.toml` (`[tools]`)
7. `.prototools`
8. `toolchain.tools` in `vis.config.ts`

That order matches what each manager reads at runtime — so `vis toolchain status` ends up reporting the same version the manager itself resolves.

## Configuration

```typescript
// vis.config.ts
import { defineConfig } from "@visulima/vis";

export default defineConfig({
    toolchain: {
        // Default: true when a manager is detected, false otherwise. When
        // true, `vis run` / `vis ci` call `<manager> install` on
        // engines.node mismatch instead of bailing.
        autoInstall: true,

        // Explicit override, useful in CI.
        preferredManager: "proto",

        // Overrides engines/packageManager-derived pins.
        tools: {
            node: ">=22.13",
            pnpm: "10.32.1",
        },
    },
});
```

## Manager compatibility

| Manager | Install command (vis maps to) | Use command (vis maps to)                    | Config file touched                    |
| ------- | ----------------------------- | -------------------------------------------- | -------------------------------------- |
| proto   | `proto install`               | `proto pin <tool> <version>`                 | `.prototools`                          |
| mise    | `mise install`                | `mise use -- <tool>@<version>`               | `.mise.toml`                           |
| fnm     | `fnm install [<version>]`     | `fnm use <version>` (Node only)              | `.nvmrc` / `.node-version`             |
| volta   | `volta install <tool>@<ver>`  | `volta pin <tool>@<version>`                 | `package.json` (`volta.<tool>` field)  |
| asdf    | `asdf install`                | `asdf local <tool> <version>`                | `.tool-versions`                       |
| nvm     | (manual)                      | (manual — nvm is a shell function)           | `.nvmrc` (edit by hand)                |

vis never edits these files itself — it shells out to the manager so the change integrates with shell hooks and caches the way the manager expects.

## Why not embed a runtime manager?

Most developers already have `fnm`, `volta`, `mise`, or `proto` installed. Shipping another ~20 MB binary would violate the principle of least surprise and duplicate behaviour the OS-level manager already handles (PATH shims, global caches, shell activation). Delegation keeps vis small and portable.

When no manager is installed, `vis toolchain status` still reports the pinned versions and their sources — the same information `vis doctor` surfaces — so the user can install `proto` or `fnm` manually and re-run.
