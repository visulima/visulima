# Custom Log Types

Pail allows you to create custom log types with custom badges, colors, labels, and log levels. This is perfect for domain-specific logging needs.

## Creating Custom Types

### Basic Custom Type

Create a logger with a custom type:

```typescript
import { createPail } from "@visulima/pail";

const pail = createPail({
    types: {
        deploy: {
            badge: "üöÄ",
            color: "blue",
            label: "deploy",
            logLevel: "informational",
        },
    },
});

pail.deploy("Application deployed successfully");
// Output: üöÄ deploy Application deployed successfully
```

## Type Configuration

### Required Properties

Each custom type requires these properties:

```typescript
interface LoggerConfiguration {
    // Visual badge/icon (can be emoji or text)
    badge?: string;
    
    // Color for the label
    color?: AnsiColors;
    
    // Text label for the log type
    label: string;
    
    // RFC 5424 log level
    logLevel: "trace" | "debug" | "informational" | "notice" | 
              "warning" | "error" | "critical" | "alert" | "emergency";
}
```

### Available Colors

Use any of these color names:

- `black`, `red`, `green`, `yellow`, `blue`, `magenta`, `cyan`, `white`
- `gray`, `grey`
- `redBright`, `greenBright`, `yellowBright`, `blueBright`, `magentaBright`, `cyanBright`, `whiteBright`

```typescript
const pail = createPail({
    types: {
        custom: {
            badge: "‚ú®",
            color: "magentaBright",
            label: "custom",
            logLevel: "informational",
        },
    },
});
```

## Overriding Default Types

You can override the default log types:

```typescript
import { createPail } from "@visulima/pail";

const pail = createPail({
    types: {
        // Override default error type
        error: {
            badge: "!!",
            color: "red",
            label: "fatal error",
            logLevel: "error",
        },
        // Override default success type
        success: {
            badge: "++",
            color: "greenBright",
            label: "huge success",
            logLevel: "informational",
        },
    },
});

pail.error("This uses the custom error type");
pail.success("This uses the custom success type");
```

## Advanced Usage

### Conditional Types

Create types based on environment:

```typescript
import { createPail } from "@visulima/pail";

const isDevelopment = process.env.NODE_ENV === "development";

const pail = createPail({
    types: {
        perf: {
            badge: isDevelopment ? "‚ö°" : "P",
            color: "yellow",
            label: isDevelopment ? "performance" : "perf",
            logLevel: "debug",
        },
    },
});
```

### Factory Function

Create reusable type configurations:

```typescript
function createServiceType(serviceName: string, emoji: string) {
    return {
        badge: emoji,
        color: "blue" as const,
        label: serviceName,
        logLevel: "informational" as const,
    };
}

const pail = createPail({
    types: {
        userService: createServiceType("user-service", "üë§"),
        orderService: createServiceType("order-service", "üì¶"),
        paymentService: createServiceType("payment-service", "üí≥"),
    },
});
```

### Type Composition

```typescript

```

## Best Practices

1. **Be meaningful** - Choose badges and labels that clearly represent the log type
2. **Stay consistent** - Use similar patterns across your application
3. **Consider accessibility** - Not all terminals display emojis well
4. **Use appropriate levels** - Map custom types to correct log levels
5. **Document types** - Keep a reference of custom types for your team
6. **Avoid overuse** - Too many custom types can be confusing
7. **Test rendering** - Verify badges display correctly in your terminal
